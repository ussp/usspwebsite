## ADDED Requirements

### Requirement: Client Document Storage

The system SHALL provide a client-scoped document store attached to the existing `clients` table via FK. Each client document SHALL have a doc_type (one of `mva`, `nda`, `ssa`, `requisition`, `work_order`, `other`), display name, description, file metadata (filename, storage path, MIME type, size), optional effective date, optional expiry date, uploader reference, free-text notes, and an optional nullable FK to `assignments.id`.

#### Scenario: Upload a client document

- **WHEN** a staff user with `client_documents.write` permission uploads a PDF, DOC, or DOCX file up to 10 MB to a client, with a doc_type and display name
- **THEN** the file is stored in the `client-documents` Supabase bucket at `{site_id}/{client_id}/{YYYY-MM}/{uuid}-{filename}`
- **AND** a `client_documents` row is written with `client_id` set, `uploaded_by` set to the current staff user, and effective_date / expiry_date null unless provided

#### Scenario: Reject disallowed file types or oversized files

- **WHEN** a staff user attempts to upload a client document whose MIME type is not PDF, DOC, or DOCX, or whose size exceeds 10 MB
- **THEN** the upload is rejected with a 400 response and an explanatory error message
- **AND** no storage object or database row is created

#### Scenario: Upload a Work Order linked to an assignment

- **WHEN** a recruiter uploads a client document with doc_type `work_order` and selects an assignment from a picker
- **THEN** the `assignment_id` FK on the new row is set to the selected assignment
- **AND** the assignment's detail page surfaces the linked Work Order in its documents section

### Requirement: Client Document Listing on Client Detail Page

The system SHALL render a "Documents" card on `/clients/[id]` that lists all `client_documents` for the client, grouped or sorted by doc_type, with upload, download, and delete actions.

#### Scenario: View client documents

- **WHEN** a staff user with `client_documents.read` permission opens `/clients/[id]`
- **THEN** the page renders a Documents card listing all non-deleted `client_documents` for that client
- **AND** each row shows doc_type label, display name, uploader, created_at, effective_date (if set), expiry_date (if set), and a Download button

#### Scenario: Client with no documents

- **WHEN** a staff user opens a client detail page for a client that has no documents
- **THEN** the Documents card renders with an empty state and an "Upload Document" call to action

### Requirement: Client Document Expiry Display

The system SHALL compute and display client document expiry status at render time based on `expiry_date`, using the same thresholds as the corporate vault (red <30 days, amber <90 days, red when expired, no badge when null).

#### Scenario: Client document expired

- **WHEN** a client document's `expiry_date` is before today
- **THEN** the row displays a red "Expired" badge with the expiry date

#### Scenario: Client document with no expiry

- **WHEN** a client document has `expiry_date = NULL`
- **THEN** the row displays no expiry badge

### Requirement: Client Documents RBAC

The system SHALL restrict client document operations to staff users with the `client_documents.*` permission. The `admin` and `recruiter` roles SHALL hold this permission. Other roles (`sales`, `hr_manager`, `viewer`) SHALL NOT.

#### Scenario: Recruiter uploads a client document

- **WHEN** a staff user with role `recruiter` uploads a client document
- **THEN** the upload succeeds subject to file validation rules
- **AND** the row's `uploaded_by` is set to the recruiter's staff user id

#### Scenario: Sales user attempts to upload a client document

- **WHEN** a staff user with role `sales` posts to the upload API
- **THEN** the API returns a 403 Forbidden response
- **AND** no database row or storage object is created

### Requirement: Client Document Download

The system SHALL provide authenticated downloads of client documents via a short-lived signed URL generated by the service client.

#### Scenario: Authorized user downloads a client document

- **WHEN** an admin or recruiter clicks "Download" on a client document
- **THEN** the server generates a signed URL for the file with a TTL of 60 seconds
- **AND** the client is redirected to the signed URL

### Requirement: Client Document Deletion

The system SHALL support deletion of a client document. Deletion SHALL remove the storage object and the database row (hard delete — no soft-delete flag in v1).

#### Scenario: Admin deletes a client document

- **WHEN** an admin clicks "Delete" on a client document and confirms the prompt
- **THEN** the storage object at the row's `storage_path` is removed from the `client-documents` bucket
- **AND** the `client_documents` row is deleted
- **AND** the UI refreshes to remove the row from the listing
